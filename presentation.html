Project Overview & Objectives

The Industry Challenge

  • Fragmented Workflows: Local restaurants struggle with disjointed online orders and config overhead.
  • Lack of Tenant Boundaries: Normal systems lack secure row-level multi-vendor data isolation.
  • Static Transactions: Customers miss live order milestones and dynamic downloadable billing receipts.

Real-Time Live Ops

Establishing instantaneous state syncs using WebSocket streams for dynamic dispatch boards and order status timelines.

Multi-Portal Architecture

Creating custom roles and views for Customers, Restaurant Admins (Vendors), and platform Super Administrators.

Enterprise Technology Stack

Frontend Ecosystem

Next.js 16/15 (App Router, React 19, TypeScript), Tailwind CSS v4 for layouts, Framer Motion for micro-animations, Recharts for analytics dashboards.

Backend Engine

NestJS 11 (Node.js framework with dependency injection and modules), Socket.IO server for bi-directional WebSockets gateway connections.

Database & Cloud

PostgreSQL 16 relational database hosted on Neon Cloud, managed via Prisma ORM for type-safe database queries and automated schema migrations.

Security & Authentication

Stateless JSON Web Tokens (JWT) featuring Access & Refresh Token Rotation. Google OAuth 2.0 social identity integration via Passport-Google.

API & Service Integrations

Razorpay checkout processing, Resend API for transactional receipt delivery, Cloudinary CDN for secure merchant media asset storage.

Global Cloud Delivery

Vercel for frontend content delivery networks (CDN) and automatic serverless deployments, Railway for scale-on-demand NestJS hosting.

System Architecture Overview

Architectural Highlights

  • Modular Backend Pattern: NestJS dependency injection manages isolated domains (Orders, Menus, Users) efficiently.
  • Stateless Security Flow: Next.js checks routing, storing JWT access tokens short-term while rotation occurs via cookie-bound refresh.
  • Neon Postgres & Prisma: Standardized multi-tenant indexing makes queries incredibly fast across vendor-specific datasets.

Cash on Delivery (COD) Razorpay Bypass

The Integration Challenge

Integrating doorstep cash payments without adding database overhead, causing state mismatch, or requiring database migrations in the relational tables.

  • Frontend Interception: Next.js checkout routes COD orders past the standard Razorpay gateway popup script.
  • Backend Branching: NestJS checkout logic skips third-party token initialization and saves orders instantly as `PENDING`.
  • Zero-Record Design Pattern: Avoids database migration by omitting payment row instantiation, tracking COD status cleanly via status tags.

    Technical Challenges & Resolutions

    CHALLENGE 1

    Timezone Analytics

    Mismatched day boundaries caused empty gaps and date inaccuracies in dashboard charts.

    Resolution:

    Standardized database operations to UTC and implemented dynamic timezone rendering client-side based on browser locations.

    CHALLENGE 2

    Tenant Data Isolation

    Preventing data access crossovers and protecting sensitive sales statistics between store vendors.

    Resolution:

    Developed custom NestJS guards to intercept incoming routes, verifying authenticated JWT token claims against query tenant targets.

    CHALLENGE 3

    Node Environment Path

    Broken global symlinks on host profiles created server start and build path blockages.

    Resolution:

    Configured explicit runtime variables in shell profiles to load localized appdata executable runtimes directly.
